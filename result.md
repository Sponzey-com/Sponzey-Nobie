# result.md

## 주제

- `analyse.md`를 바탕으로 현재 프레임워크를 어떻게 개선하면 좋은가
- 특히 워커 큐, 비동기 처리, 예약 실행, 자연어 intake 구조를 어떻게 재설계하면 좋은가

## 전제

- 자연어 해석의 중심은 계속 `LLM`이 맡는다.
- 시스템을 명령 기반 리모트 컨트롤러처럼 바꾸는 방향은 피한다.
- 개선 목표는 `자연어 처리 품질 유지`와 `응답성/안정성 향상`을 동시에 잡는 것이다.

## 현재 구조에서 핵심 문제

### 1. 접수 응답이 느리다

- 사용자가 메시지를 보내면 먼저 intake 분석이 끝나야 응답이 시작된다.
- 따라서 단순 요청도 `접수 -> 즉시 응답`이 아니라 `접수 -> 분석 완료 후 응답` 구조가 된다.

### 2. 예약 실행이 원래 태스크와 섞인다

- 예약 등록과 예약 실제 실행이 같은 흐름 안에 엮이기 쉽다.
- 그 결과 완료된 태스크가 다시 움직이는 것처럼 보이는 문제가 생긴다.

### 3. 실패 루프와 복구 루프의 역할이 섞여 있다

- intake 재분석
- 실행 복구
- 전달 복구
- 일정 등록 재시도

가 하나의 큰 루프 안에 섞이면 응답성도 나빠지고 상태 해석도 어려워진다.

### 4. 비동기 구조는 있지만 책임 분리가 약하다

- 지금도 비동기 처리와 큐는 있다.
- 하지만 `어떤 큐가 무엇을 직렬화하는지`, `어떤 작업이 새 태스크인지`, `어떤 실패가 재시도 대상인지`가 명확하게 분리되어 있지 않다.

## 권장 방향

## 1. 파이프라인을 4계층으로 분리

추천 구조는 다음과 같다.

1. `Ingress`
2. `Intake`
3. `Execution`
4. `Delivery`

### 1-1. Ingress

- Telegram/WebUI/CLI에서 요청을 받는 계층
- 여기서 해야 할 일은 최소화한다.
- 해야 하는 일:
  - 세션 식별
  - 채널 식별
  - 요청 ID 발급
  - 즉시 접수 응답
  - Intake Queue로 작업 전달

즉, Ingress는 절대 무거운 해석을 하지 않는 것이 좋다.

### 1-2. Intake

- 자연어를 해석하는 계층
- 여기서 LLM이 주 역할을 맡는다.
- 다만 결과는 구조화된 `intent envelope`로 만들어야 한다.

예시 필드:
- `intent_type`
- `target`
- `destination`
- `complete_condition`
- `execution_semantics`
- `requires_approval`
- `delivery_mode`
- `schedule_spec`

중요한 점:
- Intake는 자연어를 명령문으로 바꾸는 게 아니라
- 자연어를 실행 가능한 구조적 의미로 정리하는 역할이어야 한다.

### 1-3. Execution

- 실제 작업을 수행하는 계층
- Yeonjang 호출, 로컬 도구 실행, 스케줄 등록 등을 담당
- 이 계층은 자연어를 몰라도 되게 만드는 것이 좋다.
- Intake가 만든 구조화 결과만 보고 움직이게 해야 한다.

### 1-4. Delivery

- 결과를 어느 채널에 어떻게 전달할지 담당
- 텍스트 응답
- 파일 전달
- 텔레그램 메시지
- 화면 표시
- 예약 알림 발화

이 계층에서 처리

이렇게 나누면:
- 실행 성공과 전달 성공을 따로 볼 수 있다
- 지금처럼 “실행은 됐는데 전달이 안 됨”, “전달은 됐는데 완료 판정이 꼬임” 문제가 줄어든다.

## 2. 큐는 목적별로 나눠야 한다

현재는 큐가 있어도 의미적으로 많이 섞여 있다. 다음처럼 나누는 것이 좋다.

### 2-1. Intake Queue

- 목적: 자연어 분석 요청 직렬화/제한
- 특징:
  - 세션 기준 병렬도 제한 가능
  - 같은 사용자의 연속 요청이 들어와도 앞 요청 상태를 참고 가능
  - 하지만 무조건 request_group 직렬화와 묶을 필요는 없음

이 큐는 “해석 비용 제어”가 목적이다.

### 2-2. Execution Queue

- 목적: 실제 도구 실행 직렬화
- 특징:
  - Yeonjang 같은 외부 실행기와의 충돌 방지
  - 동일 대상 자원에 대한 동시 실행 제어
  - 권한 작업 순서 보장

이 큐는 “실행 충돌 방지”가 목적이다.

### 2-3. Delivery Queue

- 목적: 동일 채널에 대한 전달 순서 보장
- 예:
  - Telegram 같은 채널은 메시지 순서를 어지럽히지 않아야 함
  - 파일과 텍스트 전달 순서도 맞아야 함

이 큐는 “채널 응답 순서 보장”이 목적이다.

### 2-4. Schedule Queue

- 목적: 예약 실행의 독립 lifecycle 관리
- 예약 등록 요청과 예약 실제 실행은 분리해야 한다.
- 예약 실행은 원 요청 태스크와 같은 큐나 같은 request group을 쓰지 않는 편이 좋다.

이 큐는 “시간 기반 작업의 독립성”이 목적이다.

## 3. 태스크와 실행 시도를 분리

현재는 사용자가 보기엔 하나의 태스크인데 내부적으로 여러 run이 섞여 보여 혼란이 생긴다.

권장 모델:
- `Task`
- `Attempt`
- `Delivery`

### 3-1. Task

- 사용자의 의도 단위
- 예: `30초 뒤 안녕이라고 말해줘`

### 3-2. Attempt

- 그 의도를 실제로 처리하려는 실행 시도
- 예:
  - intake 시도 1
  - schedule 등록 시도 1
  - 재해석 시도 2

### 3-3. Delivery

- 최종 결과 전달 단위
- 예:
  - telegram text sent
  - telegram file sent

이렇게 분리하면:
- 완료된 태스크가 다시 실행되는 것처럼 보이는 문제를 줄일 수 있다.
- “실행은 성공했지만 전달은 실패” 같은 케이스도 명확히 관리 가능하다.

## 4. 예약 실행은 새 태스크로 보는 것이 맞다

가장 중요한 설계 포인트 중 하나다.

예약 등록 요청:
- 지금 예약을 만든다

예약 실행:
- 미래 시점에 별도의 실행이 일어난다

이 둘은 같은 태스크 그룹으로 보지 않는 편이 좋다.

권장 방식:
- 예약 등록 태스크는 등록까지만 책임진다
- 실제 실행 시점에는 새 `task instance`를 만든다
- 단, 상위 스케줄 ID나 parent task ID로 연결만 유지한다

즉:
- 같은 계보는 유지
- 같은 request group은 재사용하지 않음

이렇게 해야 중복 실행처럼 보이는 현상이 줄어든다.

## 5. 접수 응답은 즉시, 분석은 비동기로

사용자 체감 성능을 높이려면 이것이 가장 중요하다.

권장 흐름:
1. 메시지 수신
2. 세션 확인
3. 즉시 `접수됨` 응답
4. Intake Queue에 넣음
5. 분석 결과에 따라 상태 업데이트
6. 필요 시 실행/전달

이렇게 하면:
- LLM intake가 3초 걸려도 사용자는 “먹통”으로 느끼지 않는다.
- 특히 메신저 채널에서 효과가 크다.

## 6. LLM 해석과 규칙 기반 실행을 분리

자연어는 LLM이 해석해야 한다.
하지만 실행 단계까지 계속 LLM 문장을 붙들고 있으면 느리고 불안정하다.

권장:
- 자연어 해석은 LLM
- 실행은 구조화 결과 기반

즉 LLM은:
- “무슨 뜻인가”
- “무엇을 달성해야 하는가”
- “성공 조건은 무엇인가”
를 정리

그 이후에는:
- scheduler
- yeonjang executor
- delivery adapter
가 처리

## 7. 비동기 처리는 이벤트 기반으로 정리하는 편이 좋다

현재도 이벤트는 있으나 더 명시적인 event-driven 구조가 도움이 된다.

예시 이벤트:
- `task.received`
- `task.accepted`
- `intake.started`
- `intake.completed`
- `execution.started`
- `execution.failed`
- `delivery.started`
- `delivery.completed`
- `task.completed`

장점:
- 상태 모니터가 쉬워짐
- 중복 실행 추적이 쉬워짐
- 어느 단계가 병목인지 로그상 바로 보임

## 8. 재시도 예산도 분리하는 것이 좋다

지금은 재질의 예산이 한 덩어리로 느껴질 수 있다.

권장 분리:
- `intake_retry_budget`
- `execution_retry_budget`
- `delivery_retry_budget`
- `approval_wait_budget`

이렇게 나누면:
- LLM이 한 번 막혔다고 전체 작업이 과도하게 소모되지 않음
- 전달 실패와 해석 실패를 다르게 다룰 수 있음

## 9. 워커 큐는 “병렬성”보다 “책임 분리”에 초점을 둬야 한다

여기서 중요한 점은 무조건 멀티스레드로 가는 것이 아니다.

권장 생각:
- CPU 병렬화보다
- 단계별 큐 분리
- 자원 충돌 제어
- 상태 전이 명확화

즉 워커 큐는 “빠르게 돌리기 위한 병렬 프레임워크”라기보다,
“작업 단위를 안정적으로 흘려보내는 파이프라인”으로 보는 게 좋다.

## 현실적인 도입 순서

가장 안전한 순서는 다음과 같다.

### 1단계

- 접수 응답을 즉시 분리
- Intake Queue 도입
- 현재 intake 로직은 그대로 두되, 채널 응답 타이밍만 앞당김

### 2단계

- 예약 등록과 예약 실행의 task/request_group 분리
- schedule 전용 lifecycle 추가

### 3단계

- Delivery Queue 분리
- Telegram/WebUI/CLI 전달을 공통 인터페이스로 정리

### 4단계

- Execution Queue 분리
- Yeonjang/로컬 도구 충돌 제어

### 5단계

- 구조화 intent envelope를 전역 표준으로 고정
- 이후 메인 루프는 자연어가 아니라 envelope 중심으로 움직이게 함

## 최종 의견

- 자연어는 계속 LLM이 해석하는 것이 맞다.
- 따라서 프레임워크 개선의 목표는 `자연어를 명령화`하는 것이 아니라,

  `LLM 해석 결과를 안정적으로 실행/전달하는 비동기 파이프라인을 만드는 것`이어야 한다.
- 이때 필요한 것은 단순한 멀티스레드가 아니라,
  - Intake Queue
  - Execution Queue
  - Delivery Queue
  - Schedule Queue

  같은 책임 분리형 구조다.
- 가장 먼저 손대야 할 것은 성능보다 사용자 체감에 직접 영향을 주는

  `즉시 접수 응답`과 `예약 실행 분리`다.

## 10. 문서 동기화와 소멸 정책

- 구조 전환 결과는 코드와 문서가 같이 움직여야 의미가 있다.
- 고정해야 할 문서는 `process.md`, `process-to.md`, `analyse.md`, `result.md`, 현재 `.design/task00x.md`, 관련 `source.md`다.
- 구현이 먼저 바뀐 경우에는 `.design/task00x.md` 진행 메모에 차이를 먼저 기록하고, 안정화되면 `process.md/process-to.md`로 승격한다.
- 구형 코드 제거 우선순위는 다음과 같다.
  1. 사용자 화면을 왜곡하는 heuristic 보정 코드
  2. 중복 queue/route/helper
  3. execution/delivery/completion 혼합 보조 경로
  4. 임시 compat 경로
- 삭제는 replacement 경로가 기본 경로가 되고, dead-path 검색 기준과 회귀 테스트가 추가되고, build/test가 통과한 뒤에 한다.
